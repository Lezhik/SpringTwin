# AGENTS.md

Этот документ определяет правила работы AI‑агентов с репозиторием
**Spring Twin CLI**.

Spring Twin --- это CLI‑инструмент для анализа Java / Spring проектов.\
Он строит граф зависимостей, выполняет кластеризацию архитектуры и
генерирует задачи для AI‑агентов по рефакторингу.

Подробные инженерные практики, соглашения по коду и архитектурные
рекомендации описаны в **SKILLS**.\
Агенты должны руководствоваться документами из **SKILLS** при принятии
технических решений.

------------------------------------------------------------------------

# Технический стэк
- Java 21
- Spring Boot 4x
- Сборка через Gradle/Kotlin

------------------------------------------------------------------------

# Концептуальная структура проекта

Система состоит из четырёх основных этапов:

1.  **Сканирование исходников**
    -   анализ Java‑файлов
    -   извлечение зависимостей Spring DI
2.  **Анализ байткода**
    -   анализ `.class` файлов
    -   извлечение структурных зависимостей между классами
3.  **Кластеризация графа**
    -   объединение графов зависимостей
    -   группировка классов в архитектурные кластеры
4.  **Генерация задач для AI**
    -   формирование задач по рефакторингу
    -   уменьшение связности между кластерами

Каждый этап формирует JSON‑артефакт, который используется следующим
этапом.

------------------------------------------------------------------------

# Ответственность агентов

Агенты могут:

-   реализовывать новые анализаторы
-   улучшать извлечение графа зависимостей
-   улучшать алгоритмы кластеризации
-   улучшать генерацию задач для рефакторинга
-   расширять CLI функциональность

Агенты не должны:

-   ломать JSON‑форматы, используемые между этапами
-   вводить runtime‑зависимости между этапами
-   создавать сильную связанность между компонентами пайплайна

------------------------------------------------------------------------

# Артефакты пайплайна

Пайплайн работает с JSON‑файлами.

    DI graph → di.json
    Bytecode graph → bytecode.json
    Clusters → clusters.json
    Refactoring tasks → tasks.json

Форматы этих файлов должны оставаться стабильными.

------------------------------------------------------------------------

# Модель графа (высокоуровнево)

Узлы:

-   Class

Связи:

-   DEPENDS_ON
-   CALLS
-   INSTANTIATES
-   ACCESSES_FIELD

Связи содержат дополнительные детали, например:

-   тип инъекции
-   имя поля
-   имя метода

------------------------------------------------------------------------

# Философия CLI

CLI должен быть:

-   детерминированным
-   удобным для автоматизации
-   пригодным для использования в pipeline

Команды должны:

-   читать входные файлы
-   генерировать выходные файлы
-   не использовать скрытое состояние

------------------------------------------------------------------------

# Принципы проектирования

1.  Компоненты должны быть слабо связаны.
2.  Предпочтительны простые модели данных.
3.  Граф зависимостей --- центральная модель домена.
4.  Результаты анализа должны быть воспроизводимыми.

------------------------------------------------------------------------

# Best Practices

Все best practices описаны в:

    /.kilocode/skills

Агенты должны обращаться к **SKILLS** для:

-   соглашений по коду
-   архитектурных правил
-   практик тестирования
-   рекомендаций по производительности
-   шаблонов рефакторинга

------------------------------------------------------------------------

# Запрещается

Агенту запрешено без согласования с пользователем:
-   изменять структуру JSON файлов
-   изменять аргументы приложения
-   удалять или изменять уже написанные ранее автотесты
-   помещать разные классы в один файл, каждый класс должен храниться раздельно
